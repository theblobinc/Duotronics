# Duotronic v1.4 Draft Corpus

**Status:** Internal source corpus draft  
**Version:** v1.4-draft  
**Document kind:** Integrated source corpus for implementers, machine-learning systems, augmented-intelligence agents, and internal review  
**Primary purpose:** Define the current Duotronic architecture as a bridgeable, witness-gated, auto-learning representation framework that can ingest unknown or semi-known information, discover candidate patterns, synthesize candidate profiles, test those profiles, and promote only the profiles that survive validation, canonicalization, replay, migration, and policy shielding.

---

## 1. What changed in v1.4

v1.4 moves the corpus from a manually curated set of representation and witness documents into a more complete internal blueprint for a self-extending Duotronic engine.

v1.3 Draft 4 established that pronic-centered polygon families belong as a bounded geometry and witness profile. v1.4 generalizes that lesson. New representations, languages, scripts, mathematical notations, social signals, model outputs, search results, and external data streams may enter the system only through explicit evidence, witness, profile, bridge, normalizer, replay, and policy pathways.

The central addition is the **auto-profile learning layer**:

```text
raw information stream
-> evidence bundle
-> model witnesses
-> candidate segmentation
-> candidate relations
-> candidate witnesses
-> candidate profile
-> bridge proposal
-> normalizer proposal
-> replay and fixture tests
-> policy gate
-> registry staging
-> possible promotion
```

A learned profile is not trusted because a model discovered it. A learned profile becomes usable only after its object space, witness schema, canonical identity, normalizer, bridge rules, replay behavior, preservation claims, failure states, and policy boundaries are tested and registered.

---

## 2. Core thesis

Duotronics is now specified as a **presence-first, witness-gated, profile-learning representation framework**.

It is designed to work with:

1. mathematical representation families;
2. symbolic numerals such as Roman numerals;
3. visual and glyphic scripts such as hieroglyphic or Mayan-style research corpora;
4. natural language such as English;
5. polygon, graph, matrix, tensor, geometric, and symbolic objects;
6. transport rows and protocol frames;
7. search results and social-media evidence;
8. outputs from multiple machine-learning models;
9. distributed node-local observations;
10. historical lab artifacts and benchmark fixtures.

The system may learn candidate patterns from all of these. The system may not automatically treat those patterns as truth.

---

## 3. The major separation rules

Every document in this corpus preserves these separation rules:

1. **Absence is not zero.**  
   An absent object, an ordinary numeric zero, an origin role, an inactive transport row, and an invalid payload are different states.

2. **Raw witness is not canonical identity.**  
   Raw evidence can be useful, but trusted identity requires declared validation, normal-form construction, replay-stable serialization, and policy approval.

3. **Transport encoding is not semantic object.**  
   A DBP frame, Witness8 row, WSB2 sparse row, social post, video transcript, search result, OCR span, or model output is not automatically a mathematical or semantic object.

4. **Analogy is not proof.**  
   QCD, acoustics, EDO, glyphic writing, language learning, neural embeddings, and search signals may inspire profiles or diagnostics, but they do not become proofs by analogy.

5. **Learning is not promotion.**  
   A model-discovered pattern is a candidate. Promotion requires fixtures, replay, retention diagnostics, migration boundaries, registry entries, and policy authority.

## 5. Current system flow

The v1.4 trusted runtime path is:

```text
raw evidence
-> source/transport validation
-> evidence bundle
-> semantic or modality decode
-> L1 witness extraction
-> registry lookup
-> candidate/canonical normalizer selection
-> canonicalization
-> canonical identity
-> policy gate
-> L2M lookup memory
-> L2 recurrent state
-> retention diagnostics
-> replay identity check
-> L3 meta tuning
-> L4 profile or architecture proposal
-> L5 safety decision
-> promote / degrade / bypass / reject / rollback
```

The v1.4 auto-profile learning path is:

```text
unprofiled or weakly profiled stream
-> evidence bundles
-> model witnesses
-> segmentation candidates
-> relation candidates
-> witness candidates
-> profile candidate
-> bridge candidate
-> normalizer candidate
-> fixture generation
-> replay tests
-> cross-model adjudication
-> registry staging
-> sandbox runtime profile
-> possible reference or normative promotion
```

## 6. Practical examples

### Roman numerals

Roman numerals are a simple symbolic numeric adapter. A learned profile can discover symbols, subtractive patterns, canonical spelling rules, malformed forms, and a bridge into an external integer or DPFC core. The profile must declare whether import/export uses a positive-index policy or a nonnegative export policy.

### English

English is not a numeral family. It is a claim, evidence, entity, stance, source, and contradiction witness environment. The engine should not treat English text as truth. It should extract claim witnesses, source witnesses, contradiction witnesses, evidence-span witnesses, and provenance witnesses.

### Hieroglyphic or Mayan-style glyphic scripts

Glyphic scripts are multimodal visual-symbolic research profiles. The first useful witnesses are usually segmentation, visual identity, sign clustering, directionality, reading hypotheses, source alignment, and uncertainty preservation. Translation confidence should remain separate from visual canonical identity.

### Social and search streams

Search results, Reddit threads, Bluesky posts, Facebook posts, Discord messages, YouTube transcripts, comments, and platform metadata are evidence sources. They can support source, trend, claim, contradiction, timing, author, propagation, and confidence witnesses. They do not become truth without corroboration and policy authority.

---

## 7. What v1.4 is not

v1.4 is not a claim that all languages, scripts, notations, or online sources can be fully understood automatically.

v1.4 is not a claim that model consensus equals truth.

v1.4 is not a public standard.

v1.4 is not a finished production engine.

v1.4 is an internal source corpus that specifies how to build a Duotronic engine that can learn representation profiles while preserving trust, replayability, bounded authority, and policy safety.

---

## 8. Recommended implementation order

1. Implement the schema registry and profile status registry.
2. Implement evidence bundles and model witness records.
3. Implement source adapters for local documents and internal search first.
4. Implement the representation bridge interface.
5. Implement a candidate normalizer interface.
6. Implement the Roman numeral reference profile as the first auto-profile target.
7. Implement the English claim/evidence witness profile as the first natural-language profile.
8. Implement distributed node witness events.
9. Implement cross-model adjudication and contradiction tracking.
10. Implement L5 policy gates for profile promotion.
11. Implement replay, retention, and migration checks.
12. Only then allow learned profiles into sandbox runtime gating.

---

## 9. Release intent

This corpus is written for internal human review, machine-learning system review, and augmented-intelligence system utilization. It is intentionally explicit and schema-heavy. It is optimized for systems that need to read the documents and turn them into object models, tests, registries, and runtime gates.
