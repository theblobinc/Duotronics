# Corpus Review: v1.3 Draft 4 to v1.4 Draft

## 1. Structural decision

v1.4 changes the corpus from a static representation-and-witness corpus into a self-extending representation-learning corpus.

The v1.3 structure remains valid:

1. DPFC owns the mathematical and representational core.
2. The Witness Contract owns runtime trust.
3. Companion specs own registries, normalizers, transports, diagnostics, policy, migration, and lab evidence.
4. Bounded research profiles remain contained.

v1.4 adds a new middle layer: profile birth.

## 2. Why this belongs in new documents

The new material should not be scattered into every companion spec. The system now needs first-class contracts for:

1. representation bridges;
2. auto-profile learning;
3. distributed model nodes;
4. search/social evidence ingestion;
5. profile synthesis and staging.

These are broad enough to deserve owning documents.

## 3. Current v1.4 rule

The new rule is:

```text
Models may discover patterns.
Search may retrieve evidence.
Nodes may compare interpretations.
The system may synthesize profiles.
But only the Witness Contract and Policy Shield decide what can be trusted or used.
```

## 4. Practical effect

v1.4 allows the engine to learn or propose profiles for Roman numerals, English claim/evidence structure, glyphic scripts, social-source streams, external search results, graph objects, geometry objects, and other representation families.

Each learned profile must pass through evidence, witness, normalizer, bridge, replay, retention, migration, registry, and policy layers.

1. Representation Bridge Contract v1.0
2. Auto-Profile Learning Contract v1.0
3. Distributed Model Node Protocol v1.0
4. Search and Social Evidence Ingestion v1.0
5. Profile Synthesis Registry v1.0

## 6. Safety boundary

A learned profile is a candidate, not authority.

The default mode for learned profiles is audit-only. Runtime use requires fixtures, replay, retention metrics, migration/rollback planning, registry staging, and L5 policy approval.
