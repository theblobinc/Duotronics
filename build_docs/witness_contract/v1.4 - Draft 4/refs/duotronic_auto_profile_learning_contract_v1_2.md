# Duotronic Auto-Profile Learning Contract v1.2

**Status:** Research specification draft  
**Version:** auto-profile-learning-contract@v1.2  
**Supersedes:** auto-profile-learning-contract@v1.1  
**Supersedes:** auto-profile-learning-contract@v1.0  
**Document kind:** Normative learning-lifecycle contract plus reference schemas and examples  
**Primary purpose:** Define how Duotronic systems discover candidate patterns, synthesize candidate profiles, generate fixtures, compare model outputs, search for supporting evidence, and promote or reject learned profiles without bypassing witness, bridge, replay, migration, or policy rules.

---

## 1. Scope

This contract governs the birth of profiles.

It applies when the engine receives information that is:

1. unprofiled;
2. weakly profiled;
3. represented in an unknown symbolic system;
4. represented in a known but unsupported script or language;
5. represented in a visual, audio, video, graph, or multimodal form;
6. generated by an ML model;
7. retrieved from search or social feeds;
8. distributed across multiple node-local observations.

The goal is not to trust the profile immediately. The goal is to create a testable candidate.

---

## 2. Lifecycle

```text
observed_pattern
-> candidate_witness
-> candidate_profile
-> research_profile
-> sandbox_runtime_profile
-> reference_profile
-> normative_profile
```

### 2.1 Observed pattern

A recurring pattern detected in data. It may be a symbol, token, visual cluster, relation, grammar fragment, source behavior, or model output.

Allowed use: storage, clustering, comparison.

### 2.2 Candidate witness

A structured hypothesis about a pattern.

Allowed use: analysis, fixture generation, cross-model comparison.

### 2.3 Candidate profile

A proposed object space, normalizer, bridge, failure policy, and fixture set.

Allowed use: offline tests, audit-only experiments.

### 2.4 Research profile

A candidate profile with enough evidence to document and test, but not enough to influence authority.

Allowed use: sandbox research, benchmark generation.

### 2.5 Sandbox runtime profile

A research profile that may gate isolated non-authoritative ML paths.

Allowed use: controlled experiments with rollback.

### 2.6 Reference profile

A supported implementation path with replay and migration support.

Allowed use: approved bounded runtime paths.

### 2.7 Normative profile

A profile with strong evidence, stable conformance tests, policy approval, and migration support.

Allowed use: trusted runtime path.

---

## 3. Learning pipeline

```text
1. Ingest evidence bundles.
2. Decode modality without asserting semantics.
3. Run multiple model analyzers.
4. Emit model witnesses.
5. Cluster observed units.
6. Propose segmentation.
7. Propose relations.
8. Generate candidate witnesses.
9. Synthesize candidate profile.
10. Generate fixture pack.
11. Run normalizer tests.
12. Run bridge tests.
13. Run replay tests.
14. Run cross-model adjudication.
15. Search for external support if allowed.
16. Run policy gate.
17. Stage, reject, or continue observing.
```

---

## 4. Profile candidate minimum requirements

A candidate profile must define:

1. `profile_candidate_id`;
2. source evidence IDs;
3. model witness IDs;
4. discovered object space;
5. valid examples;
6. invalid examples;
7. ambiguity examples;
8. segmentation rules or object boundary rules;
9. relation rules;
10. absence policy;
11. zero policy where relevant;
12. invalid state policy;
13. canonical identity fields;
14. display-only fields;
15. normalizer candidate;
16. bridge candidate;
17. expected loss;
18. prohibited loss;
19. fixture pack;
20. replay trace set;
21. promotion request;
22. default policy mode.

If any of these are missing, the candidate may be stored as observed research material but must not become a runtime profile.

---

## 5. Model roles

The system may use multiple model roles:

| Role | Output |
|---|---|
| segmenter | candidate tokens, glyphs, clauses, image regions |
| classifier | candidate type labels |
| relation extractor | candidate relations |
| translator | candidate readings or paraphrases |
| parser inducer | candidate grammar |
| normalizer proposer | candidate canonicalization rules |
| bridge proposer | candidate source-to-target functions |
| falsifier | counterexamples and failure cases |
| search agent | external supporting or contradicting sources |
| adjudicator | agreement/disagreement summary |
| policy reviewer | risk and runtime-mode recommendation |

No role is authority by itself.

---

## 6. Fixture generation

A learned profile must generate or collect fixtures.

Fixture classes:

1. positive valid examples;
2. negative invalid examples;
3. ambiguity cases;
4. near-miss cases;
5. adversarial cases;
6. historical or dialect variants where relevant;
7. noisy OCR/ASR examples where relevant;
8. cross-source examples;
9. replay cases;
10. migration sentinel cases.

Fixtures must be pinned by hash and profile version.

---

## 7. Cross-model adjudication

Cross-model adjudication must compare:

1. segmentation agreement;
2. symbol or token inventory agreement;
3. relation agreement;
4. bridge-value agreement;
5. normalizer output agreement;
6. contradiction count;
7. source diversity;
8. model diversity;
9. replay stability;
10. policy risk.

High agreement can support staging. Low agreement can preserve uncertainty or reject.

---

## 8. Learning examples

### 8.1 Roman numerals

The engine may learn:

1. symbol inventory;
2. additive and subtractive patterns;
3. canonical spelling;
4. invalid repetition;
5. range limits;
6. no native zero;
7. numeric bridge.

### 8.2 English

The engine may learn:

1. sentence and clause boundaries;
2. entities;
3. claims;
4. quoted claims;
5. evidence spans;
6. stance;
7. contradiction relations;
8. source provenance.

It must not infer truth from grammatical confidence.

### 8.3 Glyphic scripts

The engine may learn:

1. visual region boundaries;
2. repeated glyph clusters;
3. candidate signs;
4. directionality cues;
5. candidate readings;
6. source alignments;
7. uncertainty-preserving identity.

It must not collapse visual identity and semantic translation.

---

## 9. Policy defaults

Default policy for auto-profile candidates:

```text
new candidate profile: audit_only
research profile: audit_only
sandbox runtime profile: sandbox only
reference profile: restricted or normal depending on policy
normative profile: normal if all gates pass
```

If a profile affects ML gating, memory writes, search ranking, recurrent state, or user-facing claims, it requires L5 policy approval.

---

## 10. Rejection and pruning

A learned profile should be rejected or demoted if:

1. it collapses absence and zero;
2. it cannot produce deterministic canonical identity;
3. it cannot replay profile synthesis;
4. its normalizer is unstable;
5. its bridge loses prohibited invariants;
6. it relies on a single unsupported model;
7. it mistakes search/social agreement for proof;
8. it cannot handle contradiction;
9. it fails fixture tests;
10. it cannot declare expected loss;
11. it has no migration path.

---

## 11. Non-claims

This contract does not claim autonomous understanding. It defines a disciplined process for producing, testing, and governing learned representation profiles.

## 13. Minimum fixture rule

> **Status tag:** normative

A candidate profile cannot advance to `research_profile` unless it provides at least:

1. one valid example;
2. one invalid example;
3. one ambiguity or near-miss example;
4. a fixture pack ID;
5. a replay trace set ID.

If a profile truly has no known ambiguity class, it must declare that and remain `audit_only` until the falsifier or review process accepts the omission.

---

## 14. Manual profile intake

> **Status tag:** normative

Manual profile intake uses the same `AutoProfileCandidate` shape with `origin: manual`.

A hand-authored profile may supply human rationale instead of model-discovered lineage, but it still requires:

1. fixture pack;
2. valid/invalid examples;
3. normalizer declaration;
4. bridge declaration where applicable;
5. replay trace set;
6. policy decision.

Manual profiles that lack these requirements may be stored only as `observed_pattern`, `candidate_profile`, or non-authoritative documentation.


---

## 15. Planner-triggered learning runs

> **Status tag:** normative

An auto-profile learning run may be triggered by:

1. scheduled learning policy;
2. human request;
3. profile synthesis workflow;
4. planner action candidate;
5. contradiction or uncertainty threshold;
6. source stream novelty;
7. model diversity adjudication result.

When triggered by a planner, the learning run must be linked to:

1. `DecisionContext`;
2. `ActionCandidateWitness` with `action_kind: start_profile_learning`;
3. `ProfileLearningActionPayload`;
4. `PolicyDecision`.

The learning pipeline must not infer that planner intent equals promotion authority.

### 15.1 Planner trigger result

```yaml
PlannerTriggeredLearningRun:
  planner_triggered_learning_run_id: string
  action_candidate_id: string
  decision_context_id: string
  profile_learning_action_payload_id: string
  policy_decision_id: string
  produced_candidate_profile_ids: []
  produced_candidate_witness_ids: []
  runtime_mode: audit_only | sandbox | restricted
  learning_mode: audit_only | sandbox | active
```

A planner-triggered run outputs candidate material only. Promotion requires the normal Profile Synthesis Registry path.
